# Техническое задание: Система распознавания речи с диаризацией (ASR + Diarization Service)

**Версия:** 1.0 (draft)
**Дата:** 2026-06-05
**Статус:** на согласование

---

## 1. Назначение

Система принимает аудиозаписи (встречи, звонки, интервью), выполняет:

1. Распознавание речи (ASR) с таймкодами на уровне слов.
2. Диаризацию — разбиение записи по говорящим ("кто когда говорил").
3. Идентификацию говорящих между записями через голосовые эмбеддинги и векторную БД (один и тот же человек получает стабильный ID во всех записях).
4. Выдачу готового транскрипта через REST API в нескольких форматах.

Ограничения по условию:

- **LLM не используются.** Нет суммаризации, нет генеративной пост-обработки, нет LLM-пунктуации. Пунктуация и капитализация — средствами самой ASR-модели.
- **Без облачных сервисов и Kubernetes.** Весь стек self-hosted, деплой — Docker + docker-compose на собственных серверах. Внешние API в рантайме не вызываются.

## 2. Сценарии использования

| # | Сценарий | Описание |
|---|----------|----------|
| U1 | Транскрипт встречи | Загрузка записи 30–120 мин, получение транскрипта с разметкой по спикерам |
| U2 | Обработка телефонии | Звонки 8 kHz mono/stereo; при stereo каналы = спикеры (диаризация по каналам) |
| U3 | Сквозная идентификация | "Спикер A из записи 1 = спикер C из записи 7" — через эмбеддинги в векторной БД |
| U4 | Реестр известных голосов | Регистрация эталонного голоса (имя + сэмплы); в новых записях спикер подписывается именем, а не `SPEAKER_00` |
| U5 | Поиск по архиву | Полнотекстовый поиск по транскриптам; опционально — семантический поиск по векторным представлениям сегментов |
| U6 | Экспорт | JSON / SRT / VTT / TXT, выгрузка по API |

## 3. Функциональные требования

### 3.1 Приём аудио

- **FR-1.** Загрузка файла: `multipart/form-data` либо по URL (presigned MinIO / HTTP).
- **FR-2.** Форматы: WAV, MP3, FLAC, OGG/Opus, M4A/AAC, WebM. Конвертация внутри — ffmpeg.
- **FR-3.** Лимиты (конфигурируемые): размер ≤ 2 GB, длительность ≤ 4 ч. Превышение — `413` с понятной ошибкой.
- **FR-4.** Параметры запроса: язык (`auto` | ISO-код), подсказка числа спикеров (`min/max_speakers`), webhook URL, словарь доменных терминов (initial prompt / hotwords).
- **FR-5.** Обработка асинхронная: запрос ставит задачу в очередь, мгновенно возвращает `job_id`.

### 3.2 Пайплайн обработки

```
аудио → нормализация (ffmpeg, 16kHz mono WAV)
      → VAD (отсечка тишины/музыки)
      → ASR (word-level timestamps)
      → диаризация (speaker turns)
      → выравнивание: назначение спикера каждому слову по перекрытию таймкодов
      → эмбеддинг голоса на каждый спикер-кластер
      → upsert в векторную БД + поиск ближайшего известного спикера (cosine, порог)
      → сборка транскрипта → сохранение → webhook
```

- **FR-6.** ASR: word-level timestamps обязательны; confidence на слово — желателен.
- **FR-7.** Диаризация: автоматическое определение числа спикеров; ручные `min/max_speakers` уважаются.
- **FR-8.** Stereo-телефония: режим "канал = спикер" без диаризационной модели (быстрее и точнее).
- **FR-9.** Идентификация: эмбеддинг кластера сравнивается с реестром в векторной БД; при `score ≥ threshold` (конфиг, default 0.7) — присваивается известный `speaker_id`, иначе создаётся новый анонимный.
- **FR-10.** Перекрывающаяся речь (overlap): минимум — корректные сегменты обоих спикеров, флаг `overlap: true`.
- **FR-11.** Ошибки на любом шаге не теряются: job переходит в `failed` с кодом и человекочитаемой причиной; частичный результат (например ASR без диаризации) сохраняется с флагом `partial`.

### 3.3 Векторная БД

Два набора векторов:

| Коллекция | Содержимое | Назначение |
|-----------|-----------|------------|
| `speakers` | Голосовые эмбеддинги (192–512 dim, по модели) | Идентификация / реестр голосов (U3, U4) |
| `segments` (опционально, фаза 2) | Текстовые эмбеддинги сегментов транскрипта (sentence-transformers) | Семантический поиск (U5) |

- **FR-12.** CRUD реестра спикеров: создать (имя + ≥1 аудиосэмпл ≥ 10 c чистой речи), переименовать, удалить, слить двух (merge).
- **FR-13.** Удаление спикера/записи каскадно удаляет его векторы (GDPR-совместимое right-to-erasure).

Примечание: sentence-transformers — энкодерные модели, не генеративные; ограничению "no LLM" не противоречат. Если заказчик трактует строже — коллекция `segments` исключается, остаётся полнотекстовый поиск в Postgres.

### 3.4 API

REST, OpenAPI 3.1 спецификация генерируется автоматически. Версионирование через префикс `/v1`.

| Метод | Путь | Описание |
|-------|------|----------|
| `POST` | `/v1/transcriptions` | Загрузить аудио / URL → `202 {job_id}` |
| `GET` | `/v1/jobs/{job_id}` | Статус: `queued / processing / done / failed / partial` + прогресс |
| `GET` | `/v1/transcripts/{id}` | Транскрипт, `?format=json\|srt\|vtt\|txt` |
| `GET` | `/v1/transcripts/{id}/segments` | Сегменты с пагинацией, фильтр по спикеру |
| `GET` | `/v1/transcripts` | Список с пагинацией, фильтры: дата, спикер, статус |
| `DELETE` | `/v1/transcripts/{id}` | Удаление транскрипта + аудио + векторов |
| `POST` | `/v1/speakers` | Регистрация эталонного голоса |
| `GET` | `/v1/speakers` / `/{id}` | Реестр спикеров, occurrences по записям |
| `PATCH` | `/v1/speakers/{id}` | Переименование |
| `POST` | `/v1/speakers/{id}/merge` | Слияние дублей |
| `GET` | `/v1/search?q=...` | Поиск: full-text (фаза 1), семантический (фаза 2) |
| `GET` | `/healthz`, `/readyz` | Liveness / readiness |

Формат транскрипта (JSON, основной):

```json
{
  "transcript_id": "tr_01J...",
  "audio": {"duration_sec": 1843.2, "sample_rate": 16000, "channels": 1},
  "language": "ru",
  "speakers": [
    {"local_label": "SPEAKER_00", "speaker_id": "spk_7f3a", "display_name": "Иванов И.", "match_score": 0.83},
    {"local_label": "SPEAKER_01", "speaker_id": "spk_anon_91c2", "display_name": null, "match_score": null}
  ],
  "segments": [
    {
      "id": 0, "start": 0.42, "end": 6.91,
      "speaker": "SPEAKER_00", "overlap": false,
      "text": "Добрый день, начнём со статуса по полевым работам.",
      "words": [{"w": "Добрый", "start": 0.42, "end": 0.71, "conf": 0.97}]
    }
  ]
}
```

- **FR-14.** Webhook: `POST` на указанный URL при `done/failed`, payload `{job_id, status, transcript_id}`, подпись HMAC-SHA256, ретраи 3× с экспоненциальной задержкой.
- **FR-15.** Аутентификация: API-ключи (заголовок `Authorization: Bearer`), скоупы read / write / admin. Rate limit per key.
- **FR-16.** Идемпотентность загрузки: заголовок `Idempotency-Key` — повторный запрос возвращает существующий job.

## 4. Нефункциональные требования

| Категория | Требование |
|-----------|-----------|
| Производительность | RTF ≤ 0.3 на GPU-воркере (1 ч аудио ≤ 18 мин полного пайплайна). Очередь не блокируется: API отвечает < 200 мс независимо от загрузки воркеров |
| Точность ASR | WER: ≤ 10% (чистая запись, RU/EN), ≤ 18% (телефония 8 kHz). Меряется на согласованном тест-сете до приёмки |
| Точность диаризации | DER ≤ 15% на встречах ≤ 6 спикеров; ошибка количества спикеров ≤ ±1 в 90% случаев |
| Идентификация | EER ≤ 5% на реестре до 1 000 голосов |
| Масштабирование | Горизонтальное в пределах хостов: N stateless GPU-воркеров на общей очереди; добавление воркера = запуск контейнера на ещё одном GPU-хосте, указывающего на общие Redis/Postgres/MinIO. API и воркеры деплоятся независимо |
| Надёжность | Ретрай упавшей задачи 2×; job не теряется при рестарте воркера (ack после завершения); DLQ для систематически падающих; `restart: unless-stopped` для всех контейнеров |
| Хранение | Аудио — MinIO с lifecycle-политикой (конфигурируемый TTL); транскрипты/метаданные — Postgres; векторы — Qdrant. Бэкапы: pg_dump + Qdrant snapshots + MinIO mirror, по cron |
| Безопасность | TLS на входе (reverse proxy: nginx/traefik в составе compose); шифрование at-rest (encrypted volumes / LUKS); аудио и голосовые эмбеддинги = персональные данные → удаление по запросу (FR-13), аудит-лог доступа |
| Наблюдаемость | Структурные JSON-логи с `job_id`; метрики Prometheus: глубина очереди, RTF, error rate, GPU util; Prometheus + Grafana как опциональный профиль compose |
| Офлайн-режим | Все модели скачиваются на этапе сборки образа и вшиты в него; рантайм работает без доступа в интернет |

## 5. Технологический стек

| Слой | Выбор (default) | Альтернатива | Комментарий |
|------|-----------------|--------------|-------------|
| Язык | Python 3.12+ | — | type hints обязательны, ruff + pytest |
| ASR | faster-whisper (CTranslate2, Whisper large-v3; int8_float16 на GPU) | WhisperX (даёт forced alignment wav2vec2 — точнее word timestamps) | MIT-лицензии |
| VAD | Silero VAD | pyannote VAD | |
| Диаризация | pyannote.audio 3.x (`speaker-diarization-3.1`) | NVIDIA NeMo (Sortformer/MSDD) | pyannote — gated model на HF, нужен токен на этапе сборки; лицензия MIT |
| Голосовые эмбеддинги | pyannote embedding (тот же стек) | SpeechBrain ECAPA-TDNN, NeMo TitaNet | |
| Векторная БД | Qdrant (контейнер) | pgvector (если хотят меньше компонентов) | named vectors, payload-фильтры, snapshot-бэкапы |
| API | FastAPI + uvicorn | — | pydantic v2 схемы, OpenAPI из коробки |
| Очередь | Redis + Celery (или arq) | RabbitMQ | воркер с `acks_late` |
| Метаданные | PostgreSQL 16 (контейнер) | — | full-text search (`tsvector`) для фазы 1 |
| Объектное хранилище | MinIO (контейнер) | локальный том + ручной presign-слой | S3-совместимый API, presigned URLs |
| Reverse proxy / TLS | nginx или traefik (контейнер) | — | терминация TLS, rate limit на входе |
| Деплой | Docker + docker-compose | — | GPU-воркер — отдельный образ (CUDA, `nvidia-container-toolkit`), API — CPU-образ; единый `compose.yaml` + override-файлы per-окружение |

Финальный выбор моделей фиксируется после замеров WER/DER на тест-сете заказчика (этап 0).

## 6. Архитектура

```
client ──HTTPS──> nginx/traefik ──> API (FastAPI, stateless, CPU)
                                     │  валидация, auth, presigned upload
                                     ├──> Postgres (jobs, transcripts, speakers, api_keys)
                                     ├──> MinIO (audio, экспорт-артефакты)
                                     └──> Redis (queue)
                                            │
                        GPU worker pool (Celery, контейнер на GPU-хост)
                                     │  ffmpeg → VAD → ASR → diarization → alignment → embeddings
                                     ├──> Qdrant (speakers, segments)
                                     ├──> Postgres (результат)
                                     └──> webhook → client
```

Всё — контейнеры в одном `compose.yaml` (профили: `core`, `worker`, `monitoring`). Многохостовый вариант: инфраструктурные контейнеры (Postgres, Redis, MinIO, Qdrant, API) на одном хосте, воркеры — на GPU-хостах со своим compose-файлом, подключаются по сети к общим сервисам.

Принципы:

- API не трогает модели; воркеры не принимают HTTP. Связь только через очередь и БД.
- Слоистая структура кода: `schemas/` (pydantic, формат данных) → `clients/` (обёртки MinIO/Qdrant/Postgres) → `repositories/` (intent-методы над хранилищами) → `services/` (домен: пайплайн, идентификация) → `api/` (роуты). SDK-клиенты создаются в composition root и инжектятся через `__init__`.
- Модели грузятся один раз на старте воркера (warm pool), не на задачу.

## 7. Модель данных (основное)

```
jobs:        id, status, audio_key, params(jsonb), error_code, created/started/finished_at
transcripts: id, job_id, language, duration_sec, full_text(tsvector), created_at
segments:    id, transcript_id, idx, start, end, speaker_local, speaker_id?, text, words(jsonb), overlap
speakers:    id, display_name?, kind(registered|anonymous), created_at
occurrences: speaker_id, transcript_id, local_label, match_score
api_keys:    id, hash, scopes, rate_limit, created_at, revoked_at
qdrant.speakers:  vector(emb), payload{speaker_id, source_transcript_id}
qdrant.segments:  vector(text_emb), payload{transcript_id, segment_id}   -- фаза 2
```

## 8. Этапы и приёмка

| Этап | Содержимое | Критерий приёмки |
|------|-----------|------------------|
| 0. Прототип-замер | Бенчмарк 2–3 связок ASR/диаризации на тест-сете заказчика (≥ 5 ч размеченного аудио) | Отчёт WER/DER/RTF, зафиксирован стек |
| 1. MVP | Пайплайн batch, API (upload, job, transcript JSON/SRT/TXT), Postgres+MinIO+Redis, docker-compose, auth | E2E: файл → транскрипт; нагрузочный прогон 50 параллельных задач без потерь |
| 2. Идентификация | Qdrant, реестр спикеров, сквозные ID, merge/delete, webhook | U3/U4 проходят на контрольном наборе, EER ≤ 5% |
| 3. Поиск + прод | Full-text (+семантика по решению), метрики/алерты (Prometheus+Grafana), многохостовый compose для воркеров, retention-политики, бэкапы | SLA по разделу 4, прогон на проде 2 недели без P1 |

Каждый этап: unit-тесты (клиенты хранилищ и модельные обёртки замоканы, без live-вызовов в юнитах), интеграционный smoke на стейдже (`docker compose up` на тестовом хосте), OpenAPI-спека актуальна.

## 9. Вне скоупа

- Любая генерация текста (суммаризация, action items, перевод) — LLM исключены по условию.
- Облачные managed-сервисы и Kubernetes — исключены по условию; деплой только Docker/docker-compose.
- Реал-тайм стриминг (WebSocket ASR) — отдельное ТЗ при необходимости; архитектура не должна ему противоречить (VAD и ASR-движок переиспользуемы).
- Frontend/UI — только API.
- Обучение/файнтюнинг моделей — используются предобученные.

## 10. Открытые вопросы (на согласование)

1. Языки приоритета: только RU/EN или мультиязычность? Влияет на выбор модели и тест-сет.
2. Телефония — есть ли в трафике? Если да — нужен размеченный тест-сет 8 kHz.
3. Семантический поиск (sentence-transformers) — допустим при "no LLM" или строго full-text?
4. Объём: записей/день и средняя длительность → расчёт числа GPU-воркеров.
5. Железо: какие GPU доступны (модель, VRAM, количество хостов)? large-v3 int8 требует ~5 GB VRAM; от ответа зависит выбор размера модели и параллелизм на хост.
6. TTL аудио и эмбеддингов; требования регуляторики по биометрии (голосовой эмбеддинг = биометрические ПДн в ряде юрисдикций).
