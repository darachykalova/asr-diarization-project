# Руководство пользователя и проверка по ТЗ

Дата: 2026-06-23

Документ описывает: (1) как запустить систему и открыть Swagger, (2) как пройти
проверку **каждого пункта ТЗ** прямо в Swagger, шаг за шагом.

---

## Часть 1. Запуск и доступ

### 1.1 Запустить систему

```bash
docker compose up -d
```

Проверить, что всё поднялось (должно быть 7 контейнеров, инфраструктурные — `healthy`):

```bash
docker compose ps
```

### 1.2 Открыть Swagger

В браузере (Chrome/Edge):

```
https://localhost/docs
```

- Обязательно `https`, не `http`.
- Браузер скажет **«Подключение не защищено»** (самоподписанный сертификат — это норма для localhost).
  Нажми **«Дополнительно» → «Перейти на localhost (небезопасно)»**.
  Если кнопки нет — кликни в страницу и набери вслепую `thisisunsafe`.

Альтернатива — ReDoc (только чтение): `https://localhost/redoc`.

### 1.3 Получить admin-ключ (нужен один раз)

Все защищённые методы требуют заголовок `Authorization: Bearer <ключ>`. Первый
admin-ключ создаётся напрямую в БД (эндпоинт `/v1/api-keys` сам требует admin —
курица и яйцо):

```bash
HASH=$(echo -n "test-admin-key-123456789" | sha256sum | cut -d' ' -f1)
docker exec asr_diarization_project-postgres-1 psql -U asr_user -d asr_db \
  -c "INSERT INTO api_keys (key_hash, scopes, rate_limit, created_at) VALUES ('$HASH', 'admin', 1000, NOW());"
```

Готовый тестовый ключ: **`test-admin-key-123456789`** (scope `admin` = read+write+admin).

### 1.4 Авторизоваться в Swagger

1. Вверху страницы Swagger нажми кнопку **`Authorize`** (с замочком).
2. В поле введи **только ключ** (без слова Bearer): `test-admin-key-123456789`
3. Нажми **Authorize → Close**.

Теперь все запросы из Swagger пойдут с твоим ключом. Можно тестировать.

> Как пользоваться методом в Swagger: раскрой эндпоинт → **`Try it out`** →
> заполни поля → **`Execute`** → смотри **Code** (HTTP-статус) и **Response body**.

## Часть 2. Чек-лист проверки по ТЗ

Для тестов нужен аудиофайл. Возьми любой короткий `.mp3`/`.wav` с речью
(например `data/input/audio.mp3`).

### Приём аудио

- [ ] **FR-1. Загрузка файла (multipart).**
  `POST /v1/transcriptions/upload` → Try it out → выбери файл → Execute.
  ✅ Ожидается **HTTP 202** и тело `{job_id, status:"queued", input_audio}`.

- [ ] **FR-1. Загрузка по URL.**
  `POST /v1/transcriptions/url` → тело `{"audio_url": "https://.../sample.mp3"}` → Execute.
  ✅ **HTTP 202** и `job_id`.

- [ ] **FR-2. Поддержка форматов / отказ при неверном.**
  Загрузи `.txt` или `.pdf` через `/upload`.
  ✅ Ожидается **HTTP 415** (Unsupported Media Type).

- [ ] **FR-3. Лимит размера/длительности.**
  Файл > 2 GB или длиннее 4 часов → ✅ **HTTP 413**. (На обычном файле — проходит.)

- [ ] **FR-4. Параметры запроса.**
  В `/upload` заполни `language` (напр. `ru`), `min_speakers`, `max_speakers`,
  `initial_prompt`, `webhook_url`. ✅ Принимаются, задача стартует.

- [ ] **FR-5. Асинхронность.**
  Ответ на загрузку приходит **сразу** с `job_id`, не дожидаясь обработки. ✅

### Пайплайн и результат

- [ ] **FR-5/прогресс. Статус задачи.**
  `GET /v1/jobs/{job_id}` → вставь `job_id` → Execute несколько раз.
  ✅ Статус идёт `queued → processing → done`, поле `progress` растёт 0→100.
  ✅ Несуществующий `job_id` → **HTTP 404**.

- [ ] **FR-6. Слова с таймкодами (обязательно).**
  `GET /v1/transcripts/{job_id}` после завершения.
  ✅ В каждом сегменте есть массив `words` вида
  `{"w":"…","start":…,"end":…,"conf":…}`.

- [ ] **FR-7. Диаризация.**
  В том же ответе у сегментов поле `speaker` = `SPEAKER_00`, `SPEAKER_01`, …
  ✅ Несколько говорящих → несколько меток. `min/max_speakers` уважаются.

- [ ] **FR-9. Идентификация спикеров.**
  В ответе `/transcripts/{job_id}` массив `speakers[]` с `speaker_id` и
  `match_score`. ✅ Один и тот же голос в разных записях получает один `speaker_id`
  (см. сценарий U3 ниже).

- [ ] **FR-10. Перекрытие речи.**
  У сегментов есть флаг `overlap` (true/false). ✅ Присутствует.

- [ ] **FR-11. Частичный результат / ошибки.**
  Если диаризация падает — задача не теряется: статус `partial`, текст сохранён,
  `error_message` заполнен. ✅ (на штатном файле — статус `done`).

### Экспорт и список (U6)

- [ ] **JSON.** `GET /v1/transcripts/{job_id}` → полный JSON. ✅
- [ ] **SRT / VTT / TXT.** `GET /v1/transcripts/{job_id}/export?format=srt`
  (и `vtt`, `txt`). ✅ Скачивается файл субтитров.
- [ ] **Сегменты с пагинацией.** `GET /v1/transcripts/{job_id}/segments?page=1&page_size=50`,
  фильтр `speaker_id`. ✅
- [ ] **Список транскриптов.** `GET /v1/transcripts?page=1` с фильтрами
  `status`, `speaker_id`. ✅
- [ ] **Удаление (FR-13, GDPR).** `DELETE /v1/transcripts/{job_id}`.
  ✅ Удаляет транскрипт + аудио (MinIO) + векторы (Qdrant). Повторный `GET` → 404.

### Реестр голосов (U4, FR-12)

- [ ] **Регистрация голоса.** `POST /v1/speakers` → `name` + файл `audio`
  (чистая речь **≥ 10 секунд**). ✅ Возвращает спикера; короче 10 с → **HTTP 400**.
- [ ] **Список / карточка.** `GET /v1/speakers`, `GET /v1/speakers/{id}`
  (с историей `occurrences` по записям). ✅
- [ ] **Переименование.** `PATCH /v1/speakers/{id}` → `{"name":"Иванов И."}`. ✅
- [ ] **Слияние дублей.** `POST /v1/speakers/{id}/merge` →
  `{"target_speaker_id": N}`. ✅ Записи переносятся, дубль удаляется.
- [ ] **Удаление спикера (FR-13).** `DELETE /v1/speakers/{id}`.
  ✅ Удаляет и голосовой вектор из Qdrant.
- [ ] **Записи спикера.** `GET /v1/speakers/{id}/recordings`. ✅

### Сквозная идентификация (U3) — главный сценарий

1. Загрузи **запись 1** с голосом человека А → дождись `done`.
2. Посмотри `GET /v1/transcripts/{job1}` → запомни `speaker_id` у SPEAKER_00.
3. Загрузи **запись 2** с тем же голосом → дождись `done`.
4. ✅ В `/transcripts/{job2}` тот же человек получает **тот же `speaker_id`**
   и `match_score ≥ 0.7`.

### Поиск (U5)

- [ ] **Полнотекстовый.** `GET /v1/search?q=<слово>&mode=keyword`. ✅
- [ ] **Семантический.** `GET /v1/search?q=<фраза>&mode=semantic`. ✅
- [ ] **Гибридный.** `mode=hybrid` (по умолчанию). ✅
  Фильтры `job_id`, `speaker`.

### Безопасность и надёжность (FR-14, 15, 16, NFR)

- [ ] **FR-15. Аутентификация.** Любой метод без ключа → **HTTP 401**.
  Нажми `Authorize` с ключом → проходит. Ключ только с `read` не сделает
  `write`-операцию → **HTTP 403**.
- [ ] **FR-15. Управление ключами (admin).** `POST /v1/api-keys`
  → `{"scopes":"read,write","rate_limit":100}` → выдаёт `raw_key` (показывается
  один раз). `GET /v1/api-keys`, `DELETE /v1/api-keys/{id}`. ✅
- [ ] **FR-16. Идемпотентность.** В `/upload` задай заголовок `Idempotency-Key`
  (в Swagger — поле/параметр). Повтори запрос с тем же ключом → ✅ вернётся
  **тот же** `job_id`, новая задача не создаётся.
- [ ] **FR-14. Webhook.** Укажи `webhook_url` при загрузке → по завершении на
  него уйдёт `POST {job_id, status, transcript_id}` с подписью HMAC-SHA256.
  (проверяется внешним приёмником, напр. webhook.site).
- [ ] **Rate limit.** Много запросов подряд одним ключом → при превышении лимита
  **HTTP 429**.

### Эксплуатация (NFR)

- [ ] **Liveness/Readiness.** `GET /healthz` → `{"status":"ok"}`;
  `GET /readyz` → `{"status":"ready"}` (проверяет Postgres + Redis).
  ⚠️ Они **в корне**, не под `/v1`.
- [ ] **Метрики.** Открой `https://localhost/metrics` — формат Prometheus.
- [ ] **Мониторинг (опц.).** `docker compose --profile monitoring up -d` →
  Grafana на `http://localhost:3000` (admin/admin), Prometheus на `:9090`.
- [ ] **JSON-логи.** `docker compose logs api` — строки в JSON с `job_id`.
- [ ] **Бэкапы (опц.).** `docker compose --profile backup up -d` —
  pg_dump + снапшоты.

## Часть 3. Система идентификации спикеров — как это работает

### Что происходит внутри при загрузке аудио

Когда загружается новое аудио, система автоматически:

1. Транскрибирует речь → получает сегменты с метками `SPEAKER_00`, `SPEAKER_01`, ...
2. Для каждого спикера вырезает самый длинный фрагмент речи и извлекает **голосовой отпечаток** (192-мерный вектор через SpeechBrain ECAPA-TDNN)
3. Ищет похожий голос в базе Qdrant → если совпадение ≥ 0.70 — присваивает тот же `speaker_id`, если нет — создаёт нового анонимного спикера
4. Сохраняет факт появления спикера в записи (`occurrences`) с процентом совпадения (`match_score`)

Таким образом, один и тот же человек в разных записях всегда получает **один `speaker_id`**.

---

### Как посмотреть кто есть кто

#### Шаг 1. Список всех спикеров в системе

```
GET /v1/speakers
```

В Swagger: раскрой **`GET /v1/speakers`** → Try it out → Execute.

Ответ — список спикеров с `id`, `name`, `kind`:
```json
{
  "items": [
    {"id": 6, "name": "Unknown speaker SPEAKER_00", "kind": "anonymous"},
    {"id": 7, "name": "Unknown speaker SPEAKER_01", "kind": "anonymous"}
  ]
}
```

Пока спикеру не дано имя — он `anonymous`. Имя даётся через PATCH (см. ниже).

---

#### Шаг 2. Посмотреть во всех ли записях появляется спикер

```
GET /v1/speakers/{speaker_id}/recordings
```

В Swagger: **`GET /v1/speakers/{speaker_id}/recordings`** → вставь `id` из шага 1 → Execute.

Ответ — список всех записей (job_id), где этот спикер был распознан, с процентом совпадения:
```json
[
  {"job_id": "42f5909c-...", "local_label": "SPEAKER_00", "match_score": 0.9999},
  {"job_id": "cf8ac1cc-...", "local_label": "SPEAKER_00", "match_score": 0.9999},
  {"job_id": "135a3856-...", "local_label": "SPEAKER_00", "match_score": 0.9939}
]
```

Один спикер — много записей — это подтверждение что система правильно его узнала.

---

#### Шаг 3. Посмотреть спикеров конкретной записи

```
GET /v1/transcripts/{job_id}
```

В поле `speakers[]` ответа — все спикеры этой записи с их `speaker_id` и `match_score`:
```json
{
  "speakers": [
    {
      "local_label": "SPEAKER_00",
      "speaker_id": 6,
      "display_name": "Unknown speaker SPEAKER_00",
      "match_score": 0.9999
    },
    {
      "local_label": "SPEAKER_01",
      "speaker_id": 7,
      "display_name": "Unknown speaker SPEAKER_01",
      "match_score": 0.617
    }
  ]
}
```

---

### Как читать match_score

| Значение | Что означает |
|----------|-------------|
| `null` | Первое появление голоса в системе — не с чем было сравнивать |
| `0.70 – 0.85` | Слабое совпадение (голос похож, но не уверенно) |
| `0.85 – 0.95` | Уверенное совпадение |
| `0.95 – 1.00` | Очень высокое совпадение (тот же человек, близкие условия записи) |

Порог для совпадения: **0.70** (настраивается через `SPEAKER_MATCH_THRESHOLD` в `.env`).

---

### Как дать спикеру имя

Когда понял кто это — назови его:

```
PATCH /v1/speakers/{speaker_id}
Body: {"name": "Иванов Иван"}
```

В Swagger: **`PATCH /v1/speakers/{speaker_id}`** → Try it out → вставь `id` → в поле body введи `{"name": "Иванов Иван"}` → Execute.

После этого во всех записях где этот спикер встречался — его имя обновится на `"Иванов Иван"`.

---

### Как проверить что два голоса это один человек

1. Загрузи **запись 1** → дождись `done` → посмотри `GET /v1/transcripts/{job1}` → запомни `speaker_id` у нужного спикера (например, `6`)
2. Загрузи **запись 2** с тем же человеком → дождись `done` → посмотри `GET /v1/transcripts/{job2}`
3. Если тот же человек — у него **тот же `speaker_id`** (`6`) и `match_score ≥ 0.70`

Пример из реальных данных системы:
- Запись `42f5909c` → SPEAKER_00 → `speaker_id=6`, score=**0.9999**
- Запись `cf8ac1cc` → SPEAKER_00 → `speaker_id=6`, score=**0.9999**
- Запись `135a3856` → SPEAKER_00 → `speaker_id=6`, score=**0.9939**

Один человек, три разные записи, один `speaker_id` — система работает.

---

### Что делать если система перепутала спикеров

Если два разных человека получили один `speaker_id` — это можно исправить.

**Разделить** (пока не реализовано — будет в следующей версии).

Если один человек получил два разных `speaker_id` — слить:

```
POST /v1/speakers/{id}/merge
Body: {"target_speaker_id": N}
```

Все записи переносятся на `target_speaker_id`, дубль удаляется.

---

### Быстрые команды через curl

Если удобнее терминал, а не Swagger:

```bash
# Список всех спикеров
curl -sk https://localhost/v1/speakers \
  -H "Authorization: Bearer test-admin-key-123456789" | python -m json.tool

# Все записи спикера с id=6
curl -sk https://localhost/v1/speakers/6/recordings \
  -H "Authorization: Bearer test-admin-key-123456789" | python -m json.tool

# Спикеры конкретной записи (заменить JOB_ID)
curl -sk https://localhost/v1/transcripts/JOB_ID \
  -H "Authorization: Bearer test-admin-key-123456789" | python -m json.tool

# Дать спикеру имя
curl -sk -X PATCH https://localhost/v1/speakers/6 \
  -H "Authorization: Bearer test-admin-key-123456789" \
  -H "Content-Type: application/json" \
  -d '{"name": "Иванов Иван"}' | python -m json.tool

# Загрузка с одним спикером (быстрее в 1.7x — пропускает диаризацию)
curl -sk -X POST "https://localhost/v1/transcriptions/upload?max_speakers=1" \
  -H "Authorization: Bearer test-admin-key-123456789" \
  -F "file=@audio.mp3" | python -m json.tool
```

## Часть 5. Что НЕ реализовано (честно)

- **FR-8. Стерео-телефония** («канал = спикер») — не реализовано. Единственное
  незакрытое обязательное требование. Нужно, только если в трафике есть
  стерео-звонки.
- **GPU-образ, замеры WER/DER/EER** — требуют железа и размеченного тест-сета
  заказчика (этап 0 ТЗ).
- **Swagger офлайн** — страница `/docs` грузит интерфейс из интернета; без сети
  будет пустой. На работу API не влияет.

## Шпаргалка по статусам ответов

| Код | Значение |
|-----|----------|
| 202 | Задача принята в обработку |
| 200 | Успешный GET/результат |
| 401 | Нет/неверный API-ключ |
| 403 | Ключу не хватает scope |
| 404 | Не найдено (задача/транскрипт/спикер) |
| 413 | Файл слишком большой / длинный |
| 415 | Неподдерживаемый формат аудио |
| 422 | Неверные параметры запроса |
| 429 | Превышен rate limit |
