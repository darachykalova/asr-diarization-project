# Удаление аудиозаписей из админки — дизайн

**Дата:** 2026-07-14
**Статус:** одобрен пользователем

## Цель

Сейчас в веб-админке нет способа удалить аудиозапись — только API-ключом
через `DELETE /transcripts/{job_id}` (scope `write`), недоступный из
JWT-сессии админ-консоли. Нужно дать модераторам и супер-админам удалять
запись прямо из UI (список и карточка), с подтверждением и следом в журнале
аудита.

## Решение

### Бэкенд

Новый эндпоинт `DELETE /v1/admin/audio/{job_id}` в `api/routes/admin_audio.py`:

- Auth: `Depends(get_current_user)` — без ограничения по роли (moderator и
  super_admin оба уже управляют аудио/транскрипциями по CLAUDE.md).
- Переиспользует существующую `_delete_transcript_everywhere(job_id)` из
  `api/routes/transcripts.py` (импортируется, не дублируется) — она уже
  удаляет транскрипцию из Postgres, файл из MinIO и векторы из Qdrant.
- 404, если `_delete_transcript_everywhere` возвращает `{"deleted": False,
  "reason": "not_found"}`.
- Удаление разрешено независимо от статуса записи (`queued`, `processing`,
  `done`, `failed`) — по явному решению пользователя, гонки с активной
  celery-задачей не обрабатываем.
- После успешного удаления пишет запись в журнал аудита:
  `crud.create_access_log(db, user_id=current_user.id, job_id=job_id,
  action="delete")` — событие появится в `AuditLogPage`.

Возвращаемое тело — то же, что у `_delete_transcript_everywhere` (audio_key,
minio_deleted, qdrant_deleted, postgres_deleted), плюс `message`.

### Фронтенд

- **`AudioListPage.tsx`** — новая колонка действий в таблице, кнопка
  «Удалить» в каждой строке. Клик открывает уже существующий
  `ConfirmDialog` (`danger` вариант, как для блокировки пользователя на
  `UsersPage`). После подтверждения — `fetch(DELETE .../v1/admin/audio/{job_id})`,
  при успехе строка убирается из текущего `data.items` без перезагрузки
  всего списка (оптимистичное обновление локального состояния).
- **`AudioDetailPage.tsx`** — кнопка «Удалить» рядом с заголовком записи
  (в блоке метаданных), тот же `ConfirmDialog`. После успешного удаления —
  `navigate("/audio")` (`useNavigate` из `react-router-dom`).
- Ошибки удаления показываются тем же паттерном, что и остальные ошибки на
  этих страницах (красная плашка `bg-red-50 border-red-200 text-red-700`).
- Fetch-вызов пишется инлайн в каждом файле — в проекте нет общего
  API-клиента для admin-страниц, все страницы делают fetch напрямую; общий
  хелпер был бы первой такой абстракцией без других потребителей.

## Компоненты

- `api/routes/admin_audio.py` — новый эндпоинт `DELETE /{job_id}`.
- `api/routes/transcripts.py` — без изменений, только импорт
  `_delete_transcript_everywhere`.
- `database/crud.py` — без изменений, используется существующий
  `create_access_log`.
- `frontend/src/pages/AudioListPage.tsx` — кнопка + обработчик в строке
  таблицы.
- `frontend/src/pages/AudioDetailPage.tsx` — кнопка + обработчик в карточке.

## Тестирование

- Бэкенд: новые pytest-тесты в `tests/test_admin_audio.py` —
  - успешное удаление (мок `_delete_transcript_everywhere` возвращает
    `deleted: True`) → 200 и правильное тело ответа;
  - 404, если запись не найдена;
  - требование авторизации (401 без токена);
  - доступ разрешён обеим ролям (moderator и super_admin);
  - вызывается `crud.create_access_log` с `action="delete"`.
- Фронтенд: автотестов в проекте нет вообще — ручная проверка в браузере:
  удаление из списка (строка исчезает), удаление из карточки (переход на
  список), отмена в диалоге ничего не удаляет, ошибка сети показывает
  баннер.

## Вне рамок

- Восстановление удалённых записей (soft-delete/корзина) — не просили.
- Блокировка удаления для записей в статусе `queued`/`processing` — явно
  решили удалять всегда.
- Массовое удаление (чекбоксы + «удалить выбранные») — отдельная фича, не
  сейчас.
