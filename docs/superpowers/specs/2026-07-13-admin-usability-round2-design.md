# Админка: второй заход по юзабилити — дизайн

**Дата:** 2026-07-13
**Статус:** одобрен пользователем (все 4 пункта)

## Цель

Продолжение UX-разбора (после [[2026-07-13-admin-ux-clarity-pass-design]]).
Четыре находки из повторного прохода по всем страницам, с опорой на
принципы Nielsen Norman Group и разборы loading/empty-state паттернов
(источники — в конце).

## Ключевые решения

| Проблема | Решение | Обоснование |
|---|---|---|
| Ни на одной странице нет индикатора загрузки — пустой экран, пока грузятся данные | Новый компонент `LoadingSpinner` (простой крутящийся значок на Tailwind `animate-spin`, без новых зависимостей), показывается вместо контента при **первой** загрузке страницы (не при повторных — там уже есть текст на кнопке типа «Поиск…») | Timing-based подход: пользователь должен видеть, что что-то происходит, а не гадать |
| Блокировка пользователя и смена роли срабатывают одним кликом, без подтверждения | Новый компонент `ConfirmDialog` (модальное окно в стиле сайта — макет утверждён), подключается в `UsersPage` перед обоими действиями | Nielsen: «Error prevention» — необратимые/чувствительные действия должны переспрашивать |
| На странице звонка ссылка на запись сделана как `<a href>`, перезагружает всю страницу | Заменить на `<Link to=...>` (как везде в проекте) | Единообразие, скорость — SPA-переход вместо полной перезагрузки |
| Пустой результат поиска с активными фильтрами — тупик, не предлагает сбросить | Кнопка «Сбросить фильтры» в пустом состоянии таблицы — показывается только если хоть один фильтр активен | Empty-state паттерн: «Empty search results should never actually be empty» — направить к следующему разумному действию |

## Компоненты

- **`frontend/src/components/LoadingSpinner.tsx`** (новый) — крутящийся
  значок (SVG + `animate-spin`), опциональный текст подписи.
- **`frontend/src/components/ConfirmDialog.tsx`** (новый) — модальное окно:
  заголовок, текст, кнопки «Отмена»/подтверждение (с опцией «опасного»
  красного стиля кнопки для необратимых действий).
- **Страницы, получающие `LoadingSpinner` при первой загрузке:**
  `AudioListPage`, `CallsListPage`, `AuditLogPage`, `UsersPage`,
  `AudioDetailPage`, `CallDetailPage` (в `CallDetailPage` заодно чинится
  `<a href>` → `<Link>`).
- **`UsersPage`** дополнительно получает `ConfirmDialog` перед блокировкой
  и сменой роли.
- **Пустые состояния с кнопкой сброса:** `AudioListPage`, `CallsListPage`,
  `AuditLogPage` (в двух последних — попутно добавляется недостающая
  функция сброса фильтров, её раньше не было вовсе).

## Вне рамок

- `AnalyticsPage` — не трогаем: там 4 независимых блока данных, для неё
  нужен отдельный подход к загрузке (не в этот раз).
- Автоматический повторный поиск сразу после сброса фильтров (сейчас
  «Сбросить» просто очищает поля, не запускает новый поиск — это уже так
  и не меняем, чтобы не расширять рамки задачи).
- Другие опасные действия за пределами `UsersPage` — не найдено таких.

## Тестирование

Автотестов нет. Ручная проверка после сборки:
1. Открыть каждую из 6 страниц — короткая вспышка спиннера вместо пустого
   экрана при первой загрузке.
2. На «Пользователях» — попробовать заблокировать/сменить роль, убедиться
   что появляется окно подтверждения, «Отмена» ничего не делает, а
   подтверждение — применяет действие.
3. На странице звонка — клик по ссылке «Открыть запись…» не должен
   перезагружать вкладку браузера (проверить в DevTools Network — не
   должно быть полной перезагрузки документа).
4. На «Аудиозаписях»/«Звонках»/«Журнале аудита» — задать фильтр, который
   ничего не найдёт, убедиться что появляется кнопка «Сбросить фильтры» и
   она реально очищает поля.
