# Больше сценариев мошенничества — дизайн

**Дата:** 2026-07-17
**Статус:** одобрен пользователем

## Цель

Сейчас распознаются только 3 типа мошенничества: фейковый банк, газовая
служба, полиция/следователь ([[2026-07-09-anti-scam-agent-design]]). Реальных
схем в России гораздо больше — добавляем ещё 8 самых распространённых, чтобы
агент ловил больше типов звонков. Правило простое: чем больше покрытых
сценариев, тем шире защита, при условии что каждый новый сценарий не даёт
ложных срабатываний на невинные разговоры.

## Ключевые решения

| Вопрос | Решение | Обоснование |
|---|---|---|
| Формат новых сценариев | Тот же YAML (`key`/`name`/`threshold`/`triggers` с `phrases`/`stems`/`weight`), что у существующих трёх | `load_scenarios()` уже читает всё из `call_agent/scenarios/*.yaml` через `glob` — новый файл подхватывается без единой правки кода |
| Порог срабатывания | `threshold: 70` для всех новых, как у существующих | Единообразие; менять есть смысл только если конкретный сценарий на практике даёт много ложных срабатываний |
| Как подбираются веса триггеров | Не фиксируются жёстко в этом дизайне — уточняются в реализации через TDD (тест «уверенно пересекает порог» + тест «невинная фраза не пересекает»), как это уже было с `fake_bank` (см. регрессионный тест на реальный звонок) | Веса нельзя оценить абстрактно — только прогоняя реальные фразы через `ScamDetector`; так делали и раньше |
| Что не трогаем | `dialog_engine.py`, `persona/replies.yaml`, фронтенд, `session.py` | Реплики агента и логика диалога не завязаны на конкретный ключ сценария — это чисто аддитивная правка контента |

## Новые сценарии

Один YAML-файл на сценарий в `call_agent/scenarios/`:

1. `relative_in_trouble.yaml` — родственник в беде («мама это я», «попал в
   аварию», «меня задержали», «никому не говори», «переведи деньги срочно»)
2. `mobile_operator.yaml` — мобильный оператор («номер будет заблокирован»,
   «перевыпуск сим-карты», «код для продления/отмены заявки»)
3. `tech_support.yaml` — техподдержка/вирус («вирус на компьютере»,
   «удалённый доступ», «установите программу», «продиктуйте код с экрана»)
4. `gosuslugi.yaml` — Госуслуги/соцвыплата («аккаунт госуслуг взломан»,
   «социальная выплата», «код подтверждения из смс от госуслуг»)
5. `delivery.yaml` — почта/доставка («посылка застряла на таможне»,
   «оплатите пошлину», «назовите код курьеру/водителю»)
6. `broker_investment.yaml` — брокер/инвестиции («заработок на бирже»,
   «личный консультант», «переведите на брокерский счёт», «гарантированная
   доходность»)
7. `lottery.yaml` — лотерея/приз («вы выиграли приз», «оплатите доставку
   приза», «налог на выигрыш»)
8. `fake_employer.yaml` — работодатель/вакансия («удалённая работа»,
   «оплатите обучение/материалы», «депозит за оборудование»)

Итого сценариев станет 11.

## Принцип подбора триггеров (как у существующих)

- **Сильная точная фраза** (вес 35–45) — специфичное для схемы словосочетание,
  которое почти не встречается в обычном разговоре («попал в аварию»,
  «вирус на компьютере»).
- **Комбинация из двух стеблей** (вес 55–60) — как `код`+`смс` у банка:
  по отдельности слова безобидны, вместе — уверенный сигнал («код»+«курьер»,
  «код»+«подтвержден»).
- **Слабый общий стебель** (вес 20–30) — тематическое слово само по себе не
  доказывает мошенничество («авари», «вакан», «сим»), но добавляет баллов
  в комбинации с чем-то ещё.
- Одной невинной фразы никогда не должно хватать на пересечение порога —
  для каждого сценария будет тест на это (по образцу
  `test_innocent_code_mention_stays_undetermined`).

## Тестирование

- `tests/call_agent/test_scam_detector.py`:
  - `test_loads_three_scenarios` → переименовать в `test_loads_scenarios`,
    проверять полный набор из 11 ключей.
  - На каждый новый сценарий: минимум 1 тест «фраза/комбинация уверенно
    пересекает порог» + 1 тест «похожая, но невинная фраза не пересекает».
  - Прогнать полный набор тестов после добавления — старые тесты (банк, газ,
    полиция) не должны сломаться из-за случайного пересечения триггеров между
    сценариями (например, слово «код» встречается в нескольких файлах —
    нужно смотреть на `leading_score()`, а не просто на факт срабатывания).

## Вне рамок

- Изменение весов/триггеров существующих трёх сценариев — не трогаем.
- Реплики агента, специфичные под конкретный новый сценарий (сейчас общий
  `keep_talking`/`before_hangup` набор одинаков для всех) — не делаем, это
  отдельная возможная задача в будущем.
- Живой тест новых сценариев в браузерном симуляторе — не входит в эту
  задачу (есть отдельный отложенный живой тест по `fake_bank` из
  [[2026-07-14-call-agent-scam-detection-fixes]] — не смешиваем).
